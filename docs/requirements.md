## Требования приложения saga-outbox shop

### Общие требования

Реализовать микросервисное приложение с паттерном Orchestration SAGA и Outbox для создания заказов

Реализовать базовую аутенфикацию с помощью `access` и `refresh` JWT токенов. Базовое разграничение прав RBAC. Роли:
- STAFF - просмотр пользователей, назначение других прав (в том числе STAFF) другим пользователям   

Приложение должно состоять из 5 микросервисов:

1. Orders service
2. Product service
3. Inventories service
4. Billing service
5. User service

Архитектурные решения:

- Orchestration SAGA
- Outbox pattern
- Database per Service
- Repository - паттерн проектирования для работы с базой данных

Стек:

- Язык: Python 3.13
- Инфраструктура: Git, Docker, RabbitMQ, PostgreSQL
- Используемые библиотеки: FastAPI, asynpg, yoyo-migrations, pydantic, aio-pika

### Outbox

Проблема: во время обработки сервис может сохранить данные в своей базе данных, но не отправить в брокер сообщений
(ввиду различных проблем, такие как недоступен сервер rabbitmq). Или наоборот, сервис можно запушить сообщение в
очередь, но
не сохранить в своей базе данных. Из-за чего система будет находиться в неконсистентном состоянии.

Поэтому будет реализован outbox pattern. Вместе с основной таблицей будет создана дополнительная таблица outbox, в
которую будут сохраняться данные, необходимые для отправки в брокер сообщений. Отдельная асинхронная задача будет
вычитывать новые сообщения из outbox и отправлять в rabbit.

Таблица outbox будет создана для сервисов: orders_service, inventories_service, billing_service. Product_service и
user_service не нужен этот функционал, так как такой проблем нет (товар и пользователь сохранятся только по API)

### Orchestration SAGA

Оркестрацией занимается сервис orders_service. Должны быть реализованы компенсирующие транзакции:

- В случае, если товар был зарезервирован, то оплата не прошла, нужно отменить резервацию
- В случае, если был превышен таймаут от оплаты, то необходимо отменить резервацию товара

Детальный flow будет описан ниже при описании сервиса, визуально можно
ознакомиться выше.

### Сервисы

Более детальное описание эндпоинтов каждого сервиса можно посмотреть в документации OpenAPI через Swagger на хосту
`localhost:8000/docs`

#### 1. Orders service

Отвечает за создание заказа. Сервис поддерживает endpoint POST `/api/v1/orders/create`, который создает заказ.

Flow создания заказа:

- принимает запрос от клиента
- отправляет сообщение в rabbit, которую должен вычитать product_service и отправить ответ обратно
- вычитывает сообщения от product_service:
    - в случае ошибки отправляет результат пользователю
    - в случае успеха отправляет сообщение в rabbit, которую должен вычитать inventories_service и отправить ответ
      обратно
- вычитывает сообщения от inventories_service:
    - в случае ошибки отправляет результат пользователю
    - в случае успеха отправляет сообщения в rabbit, которую должен вычитать billing_service
- вычитывает сообщения от billing_service:
    - в случае ошибки необходимо:
        - сначала отправить сообщение в inventories_service (компенсирующая транзакция) на отмену резервации товара
        - отправить ответ пользователю
    - в случае успеха отправить сообщение пользователю

Визуальный flow создания заказа можно увидеть выше.

Реализует outbox pattern, рядом с основной таблицей есть таблица outbox, чтобы не терять данные.

#### 2. Product service

Отвечает за все товары. Поддерживает следующие endpoints:

- GET `/api/v1/products/` - список всех товаров
- POST `/api/v1/products/create` - создать товар
- GET `/api/v1/products/{product_id}` - детали товара

Более детальное описание эндпоинтов можно посмотреть в документации OpenAPI через Swagger на хосту `localhost:8000/docs`

Также он вычитывает сообщения из rabbit (на детали товара), которые послал ему orders_service, обрабатывает их и
отправляет ответ обратно.

**Примечание**: Outbox pattern не реализуется, так как данные сохраняются только через API и нет риска потери сообщений.

#### 3. Inventories service

Отвечает за резервацию товара на различных складах. Поддерживает следующие endpoints:

- GET `/api/v1/inventories/` - список всех складов
- GET `/api/v1/inventories/{inventory_id}` - список всех товаров, которые есть на конкретном складе
- POST `api/v1/inventories/{inventory_id}` - добавить товаров на склад

Также он вычитывает сообщения из очереди rabbit (на резервацию товара или отмены резервации), которые послал ему
orders_service, обрабатывает и отправляет ответ обратно.

Реализует outbox pattern, рядом с основной таблицей есть таблица outbox, чтобы не терять данные.

#### 4. Billing service

Отвечает за оплату товара. Поддерживает следующие endpoints:

- GET `/api/v1/billing/{user_id}/balance` - остаток баланса пользователя
- POST `/api/v1/billing/{user_id}/deposit` - пополнить баланс пользователя

Также он вычитывает сообщения из очереди rabbit (на списание денег), которые послал ему orders service, списывает деньги
и отправляет ответ.

Реализует outbox pattern, рядом с основной таблицей есть таблица outbox, чтобы не терять данные.

#### 5. User service

Отвечает за регистрацию пользователей, выдачу JWT токенов. Поддерживает следующие endpoints:

- POST `/api/v1/users/register/` - регистрация
- GET `api/v1/users/list` - получение списка всех пользователей. Права: `ADMIN`
- POST `api/v1/users/access_token` - получение access JWT токена
- POST `api/v1/users/refresh_token` - получение refresh JWT токена

**Примечание**: Outbox pattern не реализуется, так как данные сохраняются только через API и нет риска потери сообщений.
